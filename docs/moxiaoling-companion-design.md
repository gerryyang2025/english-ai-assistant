## 2. 目标与原则

### 2.1 目标

- 让小朋友在首页感受到**有伙伴**，而不是冷冰冰的「AI 工具框」。
- **不新增后端接口**：继续走现有 `/api/chat` 与 `submitQA` 逻辑，仅改造**信息架构、视觉与轻量交互**。
- 移动端优先：虚拟形象与对话区在窄屏上仍可一手操作、可读。

### 2.2 设计原则

- **亲切**：圆角、大字号选项、鼓励性空状态与加载文案。
- **清晰**：仍是「问单词/句子 → 得解释」，功能边界不变，避免儿童以为是无限制闲聊机器人。
- **可信**：保留「联网搜索」开关与简短说明（可平铺展示，默认开启由产品决定）。
- **无障碍**：键盘可达、`aria-live` 播报回答区域、尊重 `prefers-reduced-motion`。

## 3. 角色设定：墨小灵

建议固定为**单一虚拟伙伴**，避免多角色分散认知。

| 维度 | 建议 |
|------|------|
| 定位 | 爱英语的小伙伴，擅长用简单中文解释英文单词和短句 |
| 语气 | 称呼用户「小朋友」或「你」；回答仍可由模型生成，**系统提示词**（后端或代理层）可强调：简短分段、少用术语、可举例句 |
| 视觉关键词 | 墨水、书卷、小精灵/书童感（与「墨」字呼应）；配色与站点主色 `#4A90D9` 同系，避免高饱和刺眼 |

**与产品的关系**：对外品牌名「爱英语学习 - 墨小灵」= 产品名；首页里可简称「小灵」或「墨小灵」，与 Logo 旁文案一致。

## 4. 虚拟形象方案（可选层级）

按**实现成本**从低到高排列；**当前产品已采用纯 Lottie 矢量角色**（与 [Lottie 开放格式](https://lottie.github.io/) 生态一致），不再使用位图叠加。

### 方案 A — 纯 Lottie 角色（当前实现）

- **内容**：默认 `lottie/mascot-bot.json`（bot 形象）；备选 `lottie/mascot-character.json`（LottieFiles 包）。由 `lottie-web` 在 `#moxiaoling-lottie` 内循环播放；切换见 `lottie/README.md`。
- **状态**：`idle` / `happy` 正常速率，`thinking` 略加快（`setSpeed`），辅以 CSS 外层轻微位移与光晕；`prefers-reduced-motion: reduce` 时销毁动画并以静态占位（📚）替代。
- **维护**：替换 JSON 后保持文件名，或修改 `js/app.js` 中 `MOXIAOLING_LOTTIE_BY_KEY` 的路径映射。

### 方案 A′ — 栅格插画 / PNG（已弃用于首页形象）

- 曾用 `images/moxiaoling.png`；若仅作营销物料可保留文件，首页不再引用。

### 方案 B — 矢量 / CSS 简笔画（备选）

- SVG/CSS 单图或多态，适合极轻页面；动感弱于 Lottie。

### 方案 C — 多文件 Lottie 分镜（可选迭代）

- 可拆 `idle.json` / `talk.json` 等，在 `setMoxiaolingMascotState` 内切换 `loadAnimation`；当前为单文件 + 变速以减体积与逻辑。

### 方案 D — 简笔画 + Emoji 混合（极低成本）

- 圆形头像区 + 固定插画 + 表情符号点缀；适合快速验证文案与布局。

**不推荐首期**：实时口型驱动、摄像头 AR、全屏 3D（成本高且与当前技术栈不符）。

## 5. 首页信息架构（与 AI 单词助手集成）

### 5.1 区块顺序建议

1. **Hero**（每日 Joke）— 可保留，或副标题改为「今天也和墨小灵一起学一句」类轻引导（可选）。
2. **「墨小灵」互动区**（原 `ai-qa-section` 改版）— **核心**。
3. **功能入口卡片**（feature-cards）。
4. **今日统计**。

### 5.2 「墨小灵」互动区布局（桌面 / 移动）

```
┌─────────────────────────────────────────────┐
│  [虚拟形象]     「嗨，我是墨小灵！」          │
│   (左/上)       副标题：问我英文单词或句子…   │
│                 ┌─────────────────────┐     │
│                 │ 对话气泡 / 回答区    │     │
│                 └─────────────────────┘     │
│                 [输入框] [提问]  联网开关▼   │
└─────────────────────────────────────────────┘
```

- **窄屏**：形象在上、文案与输入在下；形象宽度约 120～160px，避免占半屏。
- **对话气泡**：用户最近一次问题可显示为小气泡（可选）；回答区沿用现有 `.qa-result` + `marked` 渲染，外层包一层「墨小灵说」样式。

### 5.3 与现有逻辑的映射

| 现有 DOM / API | 改版建议 |
|----------------|----------|
| `#qa-input` | 保留 id，或改为语义化容器内仍挂同一 id，**保证 `submitQA()` 零改动或仅改选择器一次** |
| `#qa-submit-btn`、`onclick="submitQA()"` | 保留；可增加「回车提交」若尚未有 |
| `#web-search-toggle` | 保留；默认**折叠**为「更多设置」，减少首屏干扰 |
| `#qa-loading` | 文案改为「小灵正在想…」；同时切换形象为 `thinking` |
| `#qa-result` . `qa-answer` | 展示不变；加 `aria-live="polite"`，读屏在回答出现时播报 |
| `submitQA()` | **核心逻辑不动**；可在 `finally` / 显示结果时触发 `setMascotState('idle'|'happy')` 类函数 |

## 6. 交互细节

| 场景 | 行为 |
|------|------|
| 进入首页 | 形象 `idle`，可选一次「欢迎」气泡（纯前端文案，不调 API） |
| 点击提问 | 校验空输入；`thinking` + 禁用按钮（已有） |
| 成功返回答案 | `happy` 持续 2～3s 后回 `idle`（可 CSS 动画 + `setTimeout`） |
| 失败 | 形象可保持 `idle` 或轻微 `sad`（若有三态图）；文案温和 |
| 连问多轮 | 可选：在回答区上方保留「上一问」一行小字，避免长页面迷失（二期） |

## 7. 文案示例（面向小朋友）

- 标题：**墨小灵陪你学英语**
- 说明：**把你不会的英文单词或句子打在这里，小灵会用中文讲给你听**
- 占位符：**比如：future 是什么意思？ / 用 pilot 造个句**
- 加载：**小灵正在想答案…**
- 空提交：**先写一个字再问小灵吧～**

（正式语气版本可在家长说明里保留「由 AI 生成，请家长陪同使用」等，见安全。）

---

## 8. 安全与合规（儿童向）

- **不收集**儿童额外个人信息；现有流程已满足则可仅加文案。
- 首页或折叠区**简短提示**：回答由 AI 生成，仅供参考；重要学习问题可问老师或家长。
- 后端若可配置 **system prompt**：强调适龄、禁止不当内容、鼓励学习场景（与现有 MiniMax 策略对齐，需在后端评审）。

## 9. 技术实施要点（供开发拆分）

1. **HTML**：新增 `.moxiaoling-panel` 容器，内嵌 `.mascot`（`img`/`svg`）+ 重组原 `.ai-qa-section` 子节点。
2. **CSS**：新模块独立 BEM 或前缀 `moxiao-`，避免污染全局；移动端 `@media (max-width: 768px)` 单列堆叠。
3. **JS**：`submitQA` 前后各钩子 `onQAStart` / `onQAEnd`（或内联 2 行）切换 `data-mascot-state`。
4. **资源**：SVG 放 `images/moxiaoling.svg` 或内联；多状态可用 `#state-thinking` 等 `class` 切换 `display`。
5. **测试**：无 API 时错误路径 UI；`prefers-reduced-motion` 下关闭形象缩放/弹跳动画。
